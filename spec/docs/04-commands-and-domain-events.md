# 4. Commands and Domain Events Catalog

> This document is the authoritative reference for every command and domain event in the UnoArena platform. It follows the EventStorming color convention: **Blue = Command** (an intention to change state, issued by an actor or a policy), **Orange = Domain Event** (an immutable fact that has occurred), and **Purple = Policy** (a reactive rule that listens to an event and issues a new command). All event and command names match exactly the names established in documents 02 and 05 and must not be altered by downstream consumers.

---

## 4.1 Command Schema and Conventions

### 4.1.1 Standard Command Envelope

Every command issued in UnoArena — whether from a player client, an organizer UI, a scheduled trigger, or an internal policy — is wrapped in a standard envelope. The envelope enables idempotent processing, causal tracing, and concurrency control.

```
CommandEnvelope {
  commandId        : UUID          // Globally unique. Idempotency key. Client-generated.
  commandType      : string        // Fully-qualified command name (e.g., "RG.PlayCard")
  playerId         : UUID          // Identity of the actor issuing the command.
                                   // For system/policy-issued commands, a reserved
                                   // system actor ID is used.
  sessionToken     : string        // Opaque bearer token from Identity & Session.
                                   // Validated by target context before processing.
                                   // Absent for internal system commands.
  sequenceNumber   : uint64        // Per-aggregate monotonically increasing counter.
                                   // The exact semantics depend on the target aggregate
                                   // (see Section 4.1.3). Absent for commands that do
                                   // not require ordering (e.g., RegisterPlayer).
  timestamp        : ISO-8601      // Client-side timestamp. Used for latency auditing;
                                   // not trusted for ordering. Server timestamp is
                                   // authoritative.
  correlationId    : UUID          // Saga / session / flow identifier that groups
                                   // causally related commands and events across contexts.
  payload          : object        // Command-specific fields (see per-command definitions
                                   // in Section 4.2).
}
```

### 4.1.2 Standard Event Envelope

Every domain event emitted by any bounded context is wrapped in a standard envelope. This envelope enables traceability, idempotent consumption, and cryptographic integrity verification for sensitive events.

```
EventEnvelope {
  eventId          : UUID          // Globally unique. Generated by the emitting context.
  eventType        : string        // Fully-qualified event name (e.g., "RG.CardPlayed")
  aggregateId      : UUID          // The ID of the aggregate that produced this event
                                   // (e.g., gameId, roomId, tournamentId, playerId).
  aggregateType    : string        // Aggregate class name (e.g., "Game", "Room",
                                   // "Tournament", "PlayerIdentity").
  sequenceNumber   : uint64        // The aggregate's sequence number at the time this
                                   // event was emitted. Used by consumers to detect
                                   // out-of-order delivery.
  causationId      : UUID          // The commandId (or eventId, for policy-triggered
                                   // commands) that directly caused this event.
  correlationId    : UUID          // The top-level flow identifier, propagated from the
                                   // originating command envelope.
  sourceContext    : string        // Emitting bounded context abbreviation:
                                   // IS | RG | TO | RK | SV | AL
  timestamp        : ISO-8601      // Server-side authoritative timestamp.
  payload          : object        // Event-specific fields (see per-event definitions
                                   // in Section 4.3).
  signature        : string        // HMAC-SHA256 over (eventId + aggregateId +
                                   // sequenceNumber + payload). Present only for
                                   // integrity-critical events (game state mutations,
                                   // Elo updates, advancement decisions). Verified by
                                   // the Audit & Game Log context upon receipt.
  schemaVersion    : semver        // Version of the payload schema. Enables consumers
                                   // to handle schema evolution gracefully.
}
```

### 4.1.3 Sequence Numbers

Sequence numbers serve two distinct purposes in UnoArena, and the two scopes must not be conflated:

**Per-Game Sequence Number (concurrency control within a Game aggregate):**
- Each `Game` aggregate maintains a monotonically increasing counter, starting at 1.
- Every command targeting a game (`PlayCard`, `DrawCard`, `PassTurn`, `CallUno`, `ChallengeUnoCall`) must include the `sequenceNumber` that the client believes is current.
- Before processing the command, the Game aggregate checks: `command.sequenceNumber == game.expectedSequenceNumber`. If the numbers do not match, the command is rejected with `CommandRejected` (HTTP 409 Conflict, reason: `stale_sequence_number`).
- After successfully processing a command, the aggregate increments the sequence number by 1.
- This provides optimistic concurrency control: clients reconcile their state from the SSE event stream and retry with the correct number.

**Per-Player Sequence Number (ordering of a player's own commands):**
- Within a single game, each player's commands are ordered by their own per-player sequence number.
- This allows the server to detect if a player has missed an acknowledgement and is replaying a command from an earlier state.

**Commands that do not require sequence numbers:**
- Commands that operate on aggregates with lower concurrency risk (e.g., `RegisterPlayer`, `CreateRoom`, `JoinRoom`, `RegisterForTournament`) do not require a sequence number. These aggregates enforce their invariants through other means (existence checks, status checks).

### 4.1.4 Idempotency

**Command idempotency (commandId-based):**
- Every command carrier includes a `commandId` (UUID), generated by the client.
- The receiving context maintains a short-lived idempotency ledger (keyed by `commandId`).
- If a command with a previously seen `commandId` arrives, the context returns the cached response without re-processing. This is transparent to the client: the response is identical to the original.
- The idempotency window must be at least as long as the client's maximum retry interval (recommended: 24 hours).
- This allows clients to safely retry commands on network errors without risking duplicate state mutations.

**Event idempotency (eventId-based, consumer-side):**
- All event consumers must be idempotent. Because the platform guarantees at-least-once event delivery, any consumer may receive the same event more than once.
- Consumers maintain a deduplication log keyed by `eventId`. Before processing an event, the consumer checks whether `eventId` has been seen. If yes, the event is acknowledged and discarded without processing.
- Section 4.5 details the highest-risk idempotency scenarios and their deduplication strategies.

### 4.1.5 Command Rejection

When a command cannot be processed, the context emits `CommandRejected` (for game commands) or returns a synchronous error response (for HTTP-based commands). Every rejection includes:
- `commandId`: the rejected command's ID.
- `commandType`: the rejected command type.
- `reason`: a machine-readable reason code (e.g., `stale_sequence_number`, `not_your_turn`, `room_full`).
- `httpStatus`: the applicable HTTP status code (400, 409, 403, etc.).
- `message`: a human-readable description for debugging.

#### RegisterPlayer

| Field | Value |
|-------|-------|
| **Command Name** | `RegisterPlayer` |
| **Context** | Identity & Session (IS) |
| **Issuer** | Player (unauthenticated) |
| **Target Aggregate** | `PlayerIdentity` |
| **Requires Session Token** | No |
| **Requires Sequence Number** | No |

**Payload:**
```
{
  displayName  : string    // 3-32 characters, alphanumeric + underscore
  email        : string    // Valid email address
  passwordHash : string    // Hashed client-side; stored hashed again server-side
  locale       : string    // BCP-47 locale tag (optional)
}
```

**Preconditions:**
1. `displayName` is not already taken by another player.
2. `email` is not already registered.
3. `displayName` meets format constraints (length, allowed characters).
4. `email` is syntactically valid.

**Resulting Events (happy path):**
- `PlayerRegistered` -- player identity created, account state = `active`.

**Rejection Reasons:**
| Reason Code | HTTP Status | Description |
|-------------|-------------|-------------|
| `display_name_taken` | 409 | Display name already registered |
| `email_already_registered` | 409 | Email address already in use |
| `invalid_display_name` | 400 | Display name fails format validation |
| `invalid_email` | 400 | Email address is malformed |
| `password_too_weak` | 400 | Password does not meet minimum complexity |

**Idempotency Behavior:**
- If the same `commandId` is replayed, return the cached `PlayerRegistered` response.
- If a different `commandId` is submitted for an already-registered email, reject with `email_already_registered` (this is a business conflict, not a replay).

- If replayed with the same `commandId`, return the cached session token. The prior session invalidation is not re-emitted.

#### PlayCard

| Field | Value |
|-------|-------|
| **Command Name** | `PlayCard` |
| **Context** | Room Gameplay (RG) |
| **Issuer** | Player |
| **Target Aggregate** | `Game` |
| **Requires Session Token** | Yes |
| **Requires Sequence Number** | Yes (per-game) |

**Payload:**
```
{
  gameId         : UUID
  cardId         : UUID          // Unique identifier of the card in the player's hand
  declaredColor  : enum | null   // Required if card is Wild or Wild Draw Four.
                                 // One of: "red" | "blue" | "green" | "yellow"
}
```

**Preconditions:**
1. Player has a valid, active session.
2. Game is in `in_progress` state.
3. `playerId` matches the current active player (`it is your turn`).
4. `sequenceNumber` matches the game's expected next sequence number.
5. `cardId` exists in the player's hand.
6. The card is a legal play:
   a. Same color as the current active color, OR
   b. Same number / action symbol as the discard pile top card, OR
   c. The card is a Wild or Wild Draw Four (always playable — the server does not enforce the color-match restriction on Wild Draw Four; bluffing is permitted per A9. Enforcement occurs exclusively through the WDF Challenge mechanism).
7. If card is Wild or Wild Draw Four: `declaredColor` must be provided and must be one of the four valid colors.

**Resulting Events (happy path):**
- `CardPlayed` with card identity, game state changes, remaining hand size.
- Card-type-specific additional events (see Section 4.3.2: `DirectionReversed`, `PlayerSkipped`, `PenaltyCardsDrawn`, `ColorChosen`).
- If `remainingHandSize == 1` after play: `ChallengeWindowOpened`.
- If `remainingHandSize == 0` after play: `GameCompleted`.
- In all cases (unless game ended): `TurnAdvanced`.

**Rejection Reasons:**
| Reason Code | HTTP Status | Description |
|-------------|-------------|-------------|
| `invalid_session` | 401 | Session token is invalid or expired |
| `game_not_found` | 404 | Game does not exist |
| `game_not_in_progress` | 409 | Game is not in the `in_progress` state |
| `not_your_turn` | 409 | Player is not the current active player |
| `stale_sequence_number` | 409 | Sequence number mismatch (optimistic concurrency) |
| `card_not_in_hand` | 400 | Specified card is not in the player's hand |
| `illegal_play` | 400 | Card does not legally match the discard pile state |
| `declared_color_required` | 400 | Wild card played without specifying a color |
| `invalid_declared_color` | 400 | Declared color is not one of the four valid values |

**Idempotency Behavior:**
- Replaying with the same `commandId` returns the cached `CardPlayed` response. The card is not played twice. If the sequence number has since advanced, the replay check short-circuits before the sequence number check.

**Preconditions:**
1. Challenger has a valid, active session.
2. Game is in `in_progress` state.
3. A WDF Challenge Window is currently open (a Wild Draw Four was played within the last 5 seconds).
4. `challengerId` is the affected next player (the player who would draw 4 cards). No other player may issue this challenge.

**Resulting Events (happy path):**
- `WildDrawFourChallengeMade` with `gameId`, `challengerId`, `challengedPlayerId`.
- Server adjudicates by comparing the challenged player's hand snapshot against the active color at the time of the WDF play:
  - If the challenged player **held** at least one card matching the active color (bluff):
    - `WildDrawFourChallengeResolved` with `outcome = bluff_confirmed`.
    - `PenaltyCardsDrawn` for the challenged player (4 cards, `reason = wdf_challenge_bluff_confirmed`).
    - The affected player (challenger) draws 0 cards.
  - If the challenged player **did not hold** any card matching the active color (legitimate play):
    - `WildDrawFourChallengeResolved` with `outcome = legitimate_play`.
    - `PenaltyCardsDrawn` for the challenger (6 cards: 4 WDF effect + 2 penalty, `reason = wdf_challenge_legitimate_play`).
- In both cases: `WildDrawFourChallengeWindowClosed`.

**Rejection Reasons:**
| Reason Code | HTTP Status | Description |
|-------------|-------------|-------------|
| `invalid_session` | 401 | Session token is invalid or expired |
| `no_wdf_challenge_window_open` | 409 | No WDF challenge window is active |
| `wdf_window_expired` | 409 | The 5-second WDF challenge window has elapsed |
| `not_the_affected_player` | 403 | Challenger is not the player affected by the Wild Draw Four |

**Idempotency Behavior:**
- Replaying with the same `commandId` returns the cached WDF challenge resolution. The challenge is not re-resolved.

#### PlayerForfeited

| Field | Value |
|-------|-------|
| **Event Name** | `PlayerForfeited` |
| **Context** | RG |
| **Triggering Command** | `ForfeitGame`, `AutoForfeit` |
| **Triggering Policy** | `AutoForfeit` (reconnection timeout); `SessionInvalidated` → timer → `AutoForfeit` |
| **Aggregate** | `Game` |

**Key Payload Fields:**
```
{
  gameId       : UUID
  playerId     : UUID
  roomId       : UUID
  tournamentId : UUID | null
  reason       : enum    // "reconnection_timeout" | "explicit_forfeit"
                         // | "session_invalidated" | "turn_timer_expired"
  forfeitedAt  : ISO-8601
}
```

**Note on `reason` values:**
- `reconnection_timeout`: Player was disconnected and the 60-second window expired.
- `explicit_forfeit`: Player issued `ForfeitGame` directly.
- `session_invalidated`: A `SessionInvalidated` event triggered the disconnect-then-forfeit chain.
- `turn_timer_expired`: In an extreme edge case, a player's turn timer expired repeatedly and the system auto-forfeits (if configured).

**Note on card removal:** A forfeited player's cards are **removed entirely** from the game. They are NOT reshuffled back into the draw pile. This is an explicit domain rule (A36). The card removal is captured by a `HandRemoved` event emitted immediately after `PlayerForfeited`.

**Consumers and Reactions:**
| Consumer | Reaction |
|----------|----------|
| Tournament Orchestration | If `tournamentId` is present, mark `playerId` as eliminated. Record as match loss. |
| Ranking & Statistics | Increment forfeit count in player statistics (no Elo impact at this stage). |
| Spectator View | Remove player from active participants in the game projection. |
| Audit & Game Log | Append with full reason. |

**Idempotency Notes:**
- Consumers must deduplicate by `(playerId, gameId)`. A double `PlayerForfeited` for the same player in the same game would corrupt tournament elimination state. See Section 4.5.

## 4.4 Causality Chains

This section traces the key command-event-policy-command chains for the most significant domain flows. Notation:

```
CommandName [Issuer]
  → EventEmitted
    → [condition] Reaction/Policy
      → NextCommand or NextEvent
```

---

### Chain 1: Full PlayCard Chain (Including Uno and Win Mechanics)

```
PlayCard [Player]
  → CardPlayed
    → [if card is Skip]
        → PlayerSkipped (next player ID)
        → TurnAdvanced (skipping the skipped player)
    → [if card is Reverse]
        → DirectionReversed
        → TurnAdvanced (new direction applied)
        → [if 2-player game] PlayerSkipped (Reverse acts as Skip)
    → [if card is Draw Two]
        → PenaltyCardsDrawn (next player, 2 cards) [PRIVATE to player + AL]
        → PlayerSkipped (next player's turn skipped)
        → TurnAdvanced
    → [if card is Wild or Wild Draw Four]
        → ColorChosen (declared color)
        → [if Wild Draw Four]
            → PenaltyCardsDrawn (next player, 4 cards) [PRIVATE to player + AL]
            → PlayerSkipped
        → TurnAdvanced
    → [if remainingHandSize == 1 after play]
        → UnoChallengeWindowOpened (5s window)
          → Path A: CallUno [Player] within window
              → UnoCallMade
              → [if opponent issues ChallengeUnoCall after UnoCallMade]
                  → ChallengeMade
                  → ChallengeResolved (outcome: challenger_penalized)
                  → PenaltyCardsDrawn (challenger, 2 cards)
                  → UnoChallengeWindowClosed (reason: challenge_resolved)
              → [if no challenge before window closes]
                  → UnoChallengeWindowClosed (reason: expired | next_player_acted)
          → Path B: ChallengeUnoCall [Opponent] before CallUno
              → ChallengeMade
              → ChallengeResolved (outcome: target_penalized)
              → PenaltyCardsDrawn (target player, 2 cards)
              → UnoChallengeWindowClosed (reason: challenge_resolved)
          → Path C: 5 seconds elapse with no CallUno and no challenge
              → UnoChallengeWindowClosed (reason: expired)
              → [no penalty: player "gets away with" not calling]
          → Path D: Next player acts before window closes
              → UnoChallengeWindowClosed (reason: next_player_acted)
    → [if remainingHandSize == 0 after play (player played last card)]
        → GameCompleted
          → [if roomType == "casual" and not abandoned]
              → RK: ComputeEloUpdate [System policy]
                  → EloUpdated (per player)
                  → PlayerStatisticsUpdated (per player)
                  → LeaderboardUpdated (batched)
          → [if roomType == "tournament"]
              → TO: MatchResultRecorded [System policy reacting to MatchCompleted]
                  → PlayerAdvanced (top 3) | PlayerEliminated (others)
          → CheckMatchProgress policy
              → [if no player has 2 match wins yet]
                  → MatchGameCompleted
                  → StartGame [System] for next game in match
                    → (deck initialization pipeline, see Section 4.2.2)
                    → GameStarted, TurnAdvanced
              → [if a player has 2 match wins]
                  → MatchCompleted
                  → RoomCompleted
    → [in all non-game-ending cases]
        → TurnAdvanced (to next active player)
```

---

### Chain 2: Disconnection → Reconnection OR Forfeit Chain

```
[System detects heartbeat loss / SSE disconnect for Player P in Game G]
  → PlayerDisconnected (reason: connection_lost)
    → ReconnectionTimerStarted (expiresAt = now + 60s)
      → [on each subsequent TurnAdvanced where activePlayerId == P]
          → AutoSkipTurn [System]
            → TurnSkippedDueToDisconnection
            → TurnAdvanced (to next player)

      → Path A: Reconnect [Player P] within 60 seconds
          → [Reconnect command validated: valid session, window open]
            → PlayerReconnected
            → ReconnectionTimerCancelled (internal)
            → [Game continues normally; P resumes on next turn]

      → Path B: 60 seconds elapse without Reconnect
          → ReconnectionTimerExpired
            → AutoForfeit [System]
              → PlayerForfeited (reason: reconnection_timeout)
                → [if tournament room]
                    → TO receives PlayerForfeited
                    → PlayerEliminated (reason: forfeit)
                → [if only 1 active player remains]
                    → GameCompleted (winnerId = last remaining player)
                    → MatchCompleted
                    → RoomCompleted
                → [if 2+ active players remain]
                    → Game continues; P's cards removed from game

[SessionInvalidated arrives from IS for Player P]
  → RG receives SessionInvalidated
    → [if P is in an active game]
        → PlayerDisconnected (reason: session_invalidated)
          → (same timer/forfeit chain as above, but:)
            → Reconnect [Player P] will FAIL (session is invalid)
              → returns: invalid_session error
            → ReconnectionTimerExpired (inevitably after 60s)
              → AutoForfeit
                → PlayerForfeited (reason: session_invalidated)
```

---

### Chain 3: GameCompleted → Elo Update (Casual Games)

```
GameCompleted [RG emits]
  { gameId, roomType: "casual", wasAbandoned: false, placements: [...] }
  → RK policy: TriggerEloUpdate
    → [check: roomType == "casual"] PASS
    → [check: wasAbandoned == false] PASS
    → [check: gameId not in processed-games ledger] PASS (not a duplicate)
    → ComputeEloUpdate [System, internal to RK]
      → [for each player in placements]
          → Calculate Elo delta (multi-player Elo formula, position-weighted)
          → EloUpdated (playerId, gameId, oldRating, newRating, delta) [signed]
            → AL: AuditEntryRecorded
      → [for each player]
          → PlayerStatisticsUpdated
      → [batch: accumulated Elo changes feed into]
          → UpdateLeaderboard [System, periodic]
            → LeaderboardUpdated

  { gameId, roomType: "casual", wasAbandoned: true }
  → RK policy: TriggerEloUpdate
    → [check: wasAbandoned == true] FAIL
    → AbandonedGameSkipped (reason: abandoned_game)
    → [no Elo changes; statistics still updated for abandoned game count]

  { gameId, roomType: "tournament" }
  → RK policy: TriggerEloUpdate
    → [check: roomType == "casual"] FAIL
    → AbandonedGameSkipped (reason: tournament_game)
    → [no Elo changes]
```

---

### Chain 4: GameCompleted → Tournament Advancement Chain

```
GameCompleted [RG emits]
  { gameId, roomId, matchNumber, roomType: "tournament", tournamentId, roundId, placements }
  → (CheckMatchProgress policy within RG)
    → [if match not decided]
        → MatchGameCompleted
        → StartGame [System] for next game
    → [if match decided]
        → MatchCompleted
          { matchId, roomId, tournamentId, roundId, matchWins, cumulativeCardPoints,
            finalGameCompletionTime }
          → TO receives MatchCompleted
            → MatchResultRecorded (internal TO event)
          → RoomCompleted
            { roomId, roomType: "tournament", tournamentId, roundId, finalStandings }
            → TO receives RoomCompleted
              → Mark tournament room as completed
              → Rank players: top 3 by matchWins
                  → tiebreak 1: lower cumulativeCardPoints wins
                  → tiebreak 2: earlier finalGameCompletionTime wins
              → PlayerAdvanced (for positions 1, 2, 3)
              → PlayerEliminated (for positions 4+)
              → Check: all rooms in round completed?
                → [if NO: wait for remaining rooms]
                → [if YES]
                    → AllMatchesInRoundCompleted
                      → PlayersAdvanced (batch)
                      → RoundCompleted
                      → [if advancingCount > 10]
                          → CreateRound [System]
                            → TournamentRoundCreated
                            → AssignPlayersToRooms [System]
                              → RoundRoomsCreated
                              → TournamentRoomAssigned (per room)
                                → RG receives TournamentRoomAssigned
                                  → AutoJoinTournamentRoom [System]
                                    → RoomCreated, PlayerJoinedRoom (x N),
                                      RoomFilled, GameStartRequested
                                      → (deck init pipeline)
                                        → GameStarted
                      → [if advancingCount <= 10]
                          → CreateFinalRoom [System]
                            → FinalRoomCreated
                            → TournamentRoomAssigned
                              → (same RG pipeline as above)
```

---

### Chain 5: Tournament Round Completion → Next Round Creation Chain

```
[Last room in round N emits RoomCompleted]
  → TO receives RoomCompleted for the Nth room in round R
    → AllMatchesInRoundCompleted
      → RoundCompleted
        → Policy: DetermineNextPhase
          → [Count advancing players from round R]
            → [if count > 10]
                → CreateRound (roundNumber = R + 1, advancingPlayerIds)
                  → TournamentRoundCreated
                  → AssignPlayersToRooms
                    → RoundRoomsCreated
                    → TournamentRoomAssigned (per new room)
                      → RG: AutoJoinTournamentRoom (per room)
                        → RoomCreated → PlayerJoinedRoom → RoomFilled
                          → GameStartRequested → DeckInitialized
                          → InitialHandsDealt → FirstCardFlipped
                          → GameStarted → TurnAdvanced
            → [if count <= 10]
                → CreateFinalRoom (playerIds = all advancing players)
                  → FinalRoomCreated
                  → TournamentRoomAssigned (single final room)
                    → RG: AutoJoinTournamentRoom (final room)
                      → (same pipeline)
                      → [Final room plays to completion]
                        → RoomCompleted (final)
                          → TournamentCompleted
```

### Chain 6: SessionInvalidated → Forced Disconnect Chain

```
[Admin issues InvalidateSession, or Player issues Logout, or new Login occurs]
  → SessionInvalidated
    { sessionId, playerId, reason: "admin_action" | "explicit_logout" | "new_login" }
    → IS emits SessionInvalidated to event bus

    → RG consumes SessionInvalidated

        → [if player is NOT in a game]
            → No action in RG
        → [if player IS in an active game]

                → [P attempts Reconnect within window]
                    → Reconnect command REJECTED: invalid_session (401)
                    → Window continues ticking
                → [60 seconds elapse]
                    → ReconnectionTimerExpired
                    → AutoForfeit [System]

                        → [downstream: TO elimination if tournament, RK statistics, etc.]

    → AL consumes SessionInvalidated
      → AuditEntryRecorded (security forensics: session terminated)
```

## 4.5 Idempotency and Deduplication Summary

The following table enumerates all commands and events requiring special idempotency attention, ranked approximately by risk severity.

| Command / Event | Idempotency Key | Deduplication Strategy | Risk if Not Deduplicated |
|-----------------|-----------------|------------------------|--------------------------|
| `ComputeEloUpdate` (triggered by `GameCompleted`) | `gameId` | RK maintains a processed-games ledger. Before processing, check: `SELECT 1 FROM elo_processed_games WHERE game_id = ?`. If found, skip and emit `AbandonedGameSkipped(reason: already_processed)`. | **CRITICAL**: Double Elo update permanently corrupts all affected player ratings. Rating inflation or deflation affects the entire leaderboard. |
| `PlayerForfeited` consumed by TO | `(playerId, gameId)` | TO tracks forfeited players per game. Check before processing: if `playerId` already eliminated in this game, acknowledge and discard. | **HIGH**: Double elimination attempt would corrupt tournament bracket state. Player could be "eliminated twice" or advancement counts would be wrong. |
| `RoomCompleted` consumed by TO | `roomId` | TO marks rooms as `completed` atomically. Check room status before evaluating advancement. If room is already `completed`, discard event. | **HIGH**: Double round-completion check would trigger double advancement evaluation, potentially creating duplicate rounds or advancing too many players. |
| `MatchCompleted` consumed by TO | `matchId` | TO records match results keyed by `matchId`. Duplicate silently discarded after first recording. | **HIGH**: Double match result recording would double the match wins count, potentially advancing players who should not advance. |
| `SessionEstablished` | `commandId` (on `Login`) | IS uses transactional write: create new session and invalidate prior session as a single atomic operation. The prior session invalidation is conditional on the prior session existing and being active. | **HIGH**: Concurrent logins (e.g., two devices simultaneously) without atomic session management would result in two active sessions for one player, violating the single-session invariant. |
| `SessionInvalidated` consumed by RG | `(playerId, sessionId)` | RG checks if `PlayerDisconnected` has already been emitted for this player in this game. If yes, discard. | **MEDIUM**: Double `PlayerDisconnected` for the same player in the same game would start two parallel reconnection timers, causing duplicate `PlayerForfeited` events. |
| `PlayerForfeited` consumed by RK | `(playerId, gameId)` | RK checks its statistics update log. If forfeit already recorded for `(playerId, gameId)`, discard. | **MEDIUM**: Double forfeit count increment corrupts player statistics. |
| `TournamentRoomAssigned` consumed by RG | `roomId` | RG checks whether a room with `roomId` already exists. If it does, the `AutoJoinTournamentRoom` command is a no-op (idempotent room creation). | **MEDIUM**: Double room creation for the same tournament room would create two parallel game instances for the same set of players. |
| `PlayCard` / `DrawCard` / `PassTurn` | `commandId` | Game aggregate maintains idempotency ledger keyed by `commandId`. Cached response returned on replay. | **MEDIUM**: Double card play would cause a card to be played twice (same card appears twice on discard, which is physically impossible and corrupts the deck state). |
| `GameCompleted` consumed by SV | `gameId` | SV checks `projectionVersion`. If the update would regress the version, discard. | **LOW**: Double game completion event would redundantly mark the game as completed in the spectator projection. Visible artifact but no lasting data corruption. |
| `LeaderboardUpdated` | `snapshotTimestamp` | Read model consumers apply only updates with a later `snapshotTimestamp` than the current projection. | **LOW**: Double leaderboard update is idempotent if both carry the same data; at worst it causes a redundant read model refresh. |
| `AuditEntryRecorded` | `sourceEventId` | AL checks the audit trail for existing entry with `sourceEventId`. If found, discard. | **LOW**: Double audit entry is redundant but does not corrupt data. However, it inflates the audit trail and could cause confusion in replay scenarios. |

## 4.6 Cross-Context Event Contracts

This section specifies the exact data that must cross each context boundary, and the privacy constraints that govern those transfers. Consuming contexts must not modify these contracts unilaterally; changes require a versioned schema update to the publishing context's Published Language.

| Event | From | To | Required Fields | Forbidden / Privacy Constraint |
|-------|------|----|-----------------|-------------------------------|
| `GameCompleted` | RG | RK | `gameId`, `roomId`, `roomType`, `wasAbandoned`, `placements[].{playerId, position, cardPoints}`, `completedAt` | **Must NOT include hand contents.** RK does not need card identities; including them would be a privacy violation and would unnecessarily couple RK to game state internals. |
| `GameCompleted` | RG | TO | `gameId`, `roomId`, `tournamentId`, `roundId`, `roomType`, `wasAbandoned`, `placements[].{playerId, position}`, `completedAt` | TO does not need `cardPoints` (that is a RK concern). Hand contents must be absent. |
| `GameCompleted` | RG | SV | `gameId`, `roomId`, `placements[].{playerId, position}`, `completedAt` | **Card identities of any remaining hand cards MUST be stripped by the SV ACL.** Final positions are public; the specific cards held are not. |
| `GameCompleted` | RG | AL | All fields (full event including `cardPointTotals`, signed payload) | AL receives the complete, unfiltered event. This is the only context with full game state access. |
| `MatchCompleted` | RG | TO | `matchId`, `roomId`, `tournamentId`, `roundId`, `matchWinnerId` (nullable), `matchPlacements[].{playerId, position, matchWins, cumulativeCardPoints}`, `finalGameCompletionTime` | `matchPlacements` is the authoritative ranking. All three tiebreak fields (`matchWins`, `cumulativeCardPoints`, `finalGameCompletionTime`) are **mandatory** for correct advancement logic. TO must reject the event and raise an operational alert if any of these are absent. `matchWinnerId` is null in multi-player rooms (INV-M-08). |
| `MatchCompleted` | RG | SV | `matchId`, `roomId`, `matchWinnerId` (nullable), `matchPlacements[].{playerId, position, matchWins}` | `cumulativeCardPoints` may be included (it is public match data). No hand contents. |
| `RoomCompleted` | RG | TO | `roomId`, `roomType`, `tournamentId`, `roundId`, `finalStandings[].{playerId, position, matchWins, cumulativeCardPoints, completionTime}`, `outcome` | `roomType` is **mandatory** for TO to filter. TO silently ignores `RoomCompleted` events where `roomType == "casual"`. |
| `PlayerForfeited` | RG | TO | `playerId`, `gameId`, `roomId`, `tournamentId`, `reason` | `tournamentId` must be present for TO to process the elimination. TO silently ignores events where `tournamentId == null`. |
| `PlayerForfeited` | RG | RK | `playerId`, `gameId`, `roomId`, `roomType` | RK needs `roomType` to determine whether the forfeit happened in a casual or tournament context (for statistics segmentation). No card data needed. |
| `CardDrawn` | RG | SV (via ACL) | `gameId`, `playerId`, `newHandSize`, `remainingDeckSize`, `drawReason` | **`drawnCard` identity MUST be stripped (set to null) by the SV ACL.** The SV ACL is responsible for this transformation. Under no circumstances should the drawn card's identity appear in the spectator feed. |
| `CardDrawn` | RG | AL | All fields including `drawnCard` (full card identity) | AL receives the full event for game replay capability. |
| `PenaltyCardsDrawn` | RG | SV (via ACL) | `gameId`, `playerId`, `cardCount`, `reason`, `newHandSize` | **`cards` (card identity list) MUST be stripped.** Only the count is public. |
| `PenaltyCardsDrawn` | RG | AL | All fields including `cards` (full card identities) | Full event for replay. |
| `ChallengeResolved` | RG | SV (via ACL) | `gameId`, `challengerId`, `targetPlayerId`, `outcome`, `penalizedPlayerId`, `penaltyCardCount_public` | **`penaltyCards` (card identities) MUST be stripped.** Spectators see the outcome and the count, not the specific cards drawn as penalty. |
| `InitialHandsDealt` | RG | AL | All fields including all player hands (`cards` per player) | **Strictly private. Must NEVER be forwarded to SV, RK, TO, or any player other than the one who holds the hand.** AL retains for replay only. |
| `DeckInitialized` | RG | AL | All fields including `shuffleSeed` | **`shuffleSeed` is strictly private.** Must never be forwarded to SV or any client. AL retains for replay. |
| `TournamentRoomAssigned` | TO | RG | `tournamentId`, `roundId`, `roomId`, `playerIds`, `roomConfig.{maxPlayers, matchFormat, roomType}` | RG must accept this event as a command equivalent. No sensitive data crosses this boundary. |
| `EloUpdated` | RK | AL | All fields including `delta`, `oldRating`, `newRating`, plus integrity `signature` | AL must verify the signature before recording. Signed Elo events are the basis for dispute resolution. |
| `SessionInvalidated` | IS | RG | `sessionId`, `playerId`, `reason` | **No credential data (password hashes, session tokens) must cross this boundary.** Only the identity of the invalidated session is needed by RG. |
| `PlayerRegistered` | IS | RK | `playerId`, `displayName`, `registeredAt` | **`email` must NOT cross to RK.** Email is a private credential managed exclusively by IS and AL. |
| `PlayerSuspended` / `PlayerBanned` | IS | TO | `playerId`, `reason`, `adminId` | No credential data. TO needs only the player ID and the administrative action type. |
